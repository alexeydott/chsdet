# Corpus Benchmark

This directory contains a small reproducible corpus used to track detector accuracy across the main charset families supported by `chsdet`.

Files:

- `manifest.tsv`: source of truth for the corpus cases
- `data/`: byte fixtures consumed by the benchmark and the strict corpus test
- `generate-fixtures.ps1`: regenerates the fixture set
- `MATRIX.md`: latest benchmark matrix generated by `ChsDetCorpusBench.dpr`

## Tiers

- `strict`: exact `expected code page + expected name` must match; these rows are enforced by the automated test suite
- `informational`: tracked for visibility, but not enforced as a hard failure yet

## Notes

- The corpus is intentionally small and human-auditable rather than statistically broad
- The `ISO-8859-8` fixture is intentionally visual Hebrew; plain logical Hebrew is inherently ambiguous against `windows-1255` and `ISO-8859-8-I`
- The strict `windows-1253-no-c1` fixture uses non-C1 Greek discriminator bytes; byte-compatible Greek text without such bytes still remains ambiguous against `ISO-8859-7`
- The strict `cp437-nfo-art` fixture uses NFO-style pseudo-graphics so DOS/OEM art detection is checked from generated bytes rather than a one-off sample file
- The strict `ibm850`, `ibm852`, and `ibm858` fixtures use generated DOS/OEM Latin and Central European samples with byte signatures that are intentionally stronger than plain ASCII-compatible text
- The strict `windows-1250` and `koi8-u` fixtures cover conservative discriminator-byte paths rather than full language-model detection
- The informational Western rows document that Latin-1-like bytes collapse into `windows-1252`, because `ISO-8859-1` is not exposed as a separate output charset
- `x-euc-tw` is not part of the exact corpus yet because the standard Windows/.NET code-page path does not provide a trustworthy fixture source for that encoding variant
